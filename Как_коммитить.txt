====================================================================
   ИНСТРУКЦИЯ: КАК ДЕЛАТЬ КОММИТЫ В GIT (простыми словами)
====================================================================

Привет! Это шпаргалка, чтобы ты сам мог коммитить, когда захочешь.
Здесь всё разжевано на примере того, что мы уже сделали в проекте.


--------------------------------------------------------------------
ЧТО ТАКОЕ КОММИТ И ЗАЧЕМ ОН НУЖЕН
--------------------------------------------------------------------

Коммит (commit) — это как "сохранить игру". Ты делаешь слепок всех
своих файлов в один момент времени и подписываешь его: "вот, я сделал
вот это". Git хранит историю всех таких слепков.

Когда ты видишь букву "M" рядом с файлом в VS Code (обычно жёлтым или
оранжевым) — это значит:
   "Файл ИЗМЕНЁН после последнего коммита, но ещё не сохранён в Git."

Чтобы буква "M" исчезла — нужно закоммитить изменения. После коммита
файл снова "чистый".


--------------------------------------------------------------------
БАЗОВАЯ ПОСЛЕДОВАТЕЛЬНОСТЬ (3 шага)
--------------------------------------------------------------------

В терминале VS Code (Ctrl+` чтобы открыть):

   ШАГ 1. Посмотреть, что изменилось:
   ----------------------------------------------------------------
   git status --short
   ----------------------------------------------------------------
   Покажет список файлов с буквами слева:
     M  file.html       — файл изменён (Modified)
     ??  new-file.txt   — новый файл, ещё не в Git (Untracked)
     D  old-file.txt    — файл удалён (Deleted)
     A  file.txt        — уже добавлен в индекс (Added)

   ШАГ 2. Добавить изменения в "корзину коммита" (staging area):
   ----------------------------------------------------------------
   git add -A
   ----------------------------------------------------------------
   Берёт ВСЕ изменённые/новые/удалённые файлы и готовит их к коммиту.
   -A = "все" (All).

   Если хочешь добавить только один конкретный файл:
   ----------------------------------------------------------------
   git add src/index.html
   ----------------------------------------------------------------

   ШАГ 3. Сделать сам коммит с понятным сообщением:
   ----------------------------------------------------------------
   git commit -m "Короткое описание того, что сделал"
   ----------------------------------------------------------------
   Флажок -m значит "message" (сообщение). После него в кавычках —
   что ты сделал. Пиши кратко, но понятно. Например:
     "Починил калькулятор на главной"
     "Обновил цвета в подвале"
     "Добавил раздел отзывов"

   Готово! Буква "M" в VS Code исчезнет.


--------------------------------------------------------------------
ПОЛНЫЙ ПРИМЕР: ЧТО МЫ СДЕЛАЛИ В ЭТОТ РАЗ
--------------------------------------------------------------------

Ситуация: я правил файл src/css/input.css и ты просил закоммитить
всё, чтобы убрать буквы "M".

--- БЫЛО (до коммита) ----------------------------------------------

В VS Code у файлов горели буквы "M":
   M  docs/consent.html
   M  docs/disclaimer.html
   M  docs/index.html
   M  docs/privacy.html
   M  src/consent.html
   M  src/css/input.css        ← тут мои исправления
   M  src/disclaimer.html
   M  src/index.html
   M  src/privacy.html
   ?? docs/assets/input-Da8qE6BE.css    ← новый файл (не в Git)
   ?? docs/assets/main-DOIcPtqe.js      ← новый файл (не в Git)

Команда git status --short показала то же самое.


--- ЧТО Я СДЕЛАЛ ---------------------------------------------------

1) Сначала проверил, что именно изменено (чтобы не закоммитить
   случайный мусор или секреты):
   ----------------------------------------------------------------
   git diff src/index.html src/consent.html src/disclaimer.html \
             src/privacy.html src/css/input.css
   ----------------------------------------------------------------
   Увидел, что изменения адекватные: смена цветов, добавлены id-якоря,
   почищен input.css. Секретов нет.

2) Посмотрел историю, чтобы понять стиль коммитов в репо:
   ----------------------------------------------------------------
   git log --oneline -10
   ----------------------------------------------------------------
   Увидел:
     4c1f4d8 Первая заливка
   Значит, стиль сообщений — простой русский текст.

3) Добавил ВСЕ изменения в индекс:
   ----------------------------------------------------------------
   git add -A
   ----------------------------------------------------------------

4) Проверил, что попало в индекс:
   ----------------------------------------------------------------
   git status --short
   ----------------------------------------------------------------
   Слева появились буквы A/D/M/R — всё на месте.

5) Сделал коммит с понятным сообщением:
   ----------------------------------------------------------------
   git commit -m "Ребрендинг: обновление цветовой схемы и фиксы скролла; очистка input.css от мусора"
   ----------------------------------------------------------------

--- СТАЛО (после коммита) ------------------------------------------

В терминале Git ответил:
   [main 5910363] Ребрендинг: обновление цветовой схемы...
   12 files changed, 340 insertions(+), 324 deletions(-)

Команда git status --short показала ПУСТОЙ вывод — значит, всё
закоммичено, рабочее дерево чистое.

В VS Code буквы "M" у всех файлов ИСЧЕЗЛИ. Файлы стали "чистыми".


--------------------------------------------------------------------
ПОЛЕЗНЫЕ КОМАНДЫ ДЛЯ ПРОВЕРКИ
--------------------------------------------------------------------

   Посмотреть историю всех коммитов:
   ----------------------------------------------------------------
   git log --oneline -10
   ----------------------------------------------------------------
   -10 = показать последние 10. Цифру можно менять.

   Посмотреть, ЧТО ИМЕННО изменилось (до коммита):
   ----------------------------------------------------------------
   git diff
   ----------------------------------------------------------------
   Покажет построчно: зелёным — что добавилось, красным — что удалилось.

   Посмотреть только по одному файлу:
   ----------------------------------------------------------------
   git diff src/css/input.css
   ----------------------------------------------------------------

   Посмотреть сводку (сколько строк изменилось, без деталей):
   ----------------------------------------------------------------
   git diff --stat
   ----------------------------------------------------------------

   Проверить, чистое ли рабочее дерево:
   ----------------------------------------------------------------
   git status --short
   ----------------------------------------------------------------
   Если вывод пустой — всё чисто, коммитить нечего.
   Если есть строки с буквами — есть незакоммиченные изменения.


--------------------------------------------------------------------
ТИПИЧНЫЙ СЦЕНАРИЙ "Я ОТРЕДАКТИРОВАЛ ФАЙЛ, ЧТО ДАЛЬШЕ?"
--------------------------------------------------------------------

Допустим, ты поправил src/index.html. Что делать:

   1) git status --short
        → увидишь:  M  src/index.html

   2) git diff src/index.html
        → посмотришь, что именно изменилось (для себя, контроль)

   3) git add src/index.html
        → добавляешь именно этот файл в коммит

   4) git commit -m "Поправил заголовок на главной"
        → делаешь коммит

   5) git status --short
        → вывод пустой, буква "M" в VS Code исчезла. Победа!


--------------------------------------------------------------------
ВАЖНЫЕ ПРАВИЛА (читай внимательно!)
--------------------------------------------------------------------

   ✅ ПРАВИЛЬНО:
   - Коммить логически связанные изменения вместе.
     (например: "фикс калькулятора" — всё, что относится к калькулятору)
   - Пиши понятные сообщения коммита.
     Плохо:  "изменения"
     Хорошо: "Поправил отступы в подвале на мобильных"
   - Проверяй git diff перед коммитом, чтобы не попасть лишнее.
   - Коммить часто, мелкими порциями — так проще откатиться.

   ❌ НЕЛЬЗЯ / ПЛОХО:
   - Не коммить пароли, токены, API-ключи, .env файлы.
     Если случайно попало — надо срочно убрать и сделать новый коммит.
   - Не пиши коммиты без сообщения (git commit без -m откроет редактор).
   - Не коммить мусор: временные файлы, билды, логи (для этого есть .gitignore).


--------------------------------------------------------------------
ЕСЛИ ЧТО-ТО ПОШЛО НЕ ТАК
--------------------------------------------------------------------

   Случайно добавил не тот файл в индекс (ещё НЕ закоммитил):
   ----------------------------------------------------------------
   git restore --staged имя-файла
   ----------------------------------------------------------------
   Файл уйдёт из "корзины", но сами изменения останутся.

   Нужно отменить ПОСЛЕДНИЙ коммит (но сохранить изменения):
   ----------------------------------------------------------------
   git reset --soft HEAD~1
   ----------------------------------------------------------------
   Коммит исчезнет, изменения вернутся в индекс, ничего не потеряешь.

   Нужно глянуть, что было в каком-то старом коммите:
   ----------------------------------------------------------------
   git show хэш-коммита
   ----------------------------------------------------------------
   Например: git show 5910363
